# HYP-SPEC-010500

# Versioning

**Project:** Hypatia Knowledge Platform

**Document ID:** HYP-SPEC-010500

**Version:** 1.0 Founder Edition

**Status:** Official

**Classification:** Architecture Standard

---

# Purpose

This document defines the official versioning policy of the Hypatia Platform.

Every object inside Hypatia evolves.

Nothing is overwritten.

Nothing is lost.

Every scientific state remains reproducible.

---

# Fundamental Principle

Knowledge evolves.

History is preserved.

Versions are immutable.

---

# Version Hierarchy

Hypatia maintains independent version histories for different components.

A change in one component does not imply a version change in another.

---

# Version Domains

The platform maintains independent version sequences for:

```text
Platform
Repository
Specifications
Knowledge Objects
Cards
Relations
Datasets
Resolvers
Builders
Website
AI Models
```

Each domain evolves independently.

---

# Repository Version

Represents the scientific state of the complete repository.

Example

```text
Repository Version

1.0.0
```

This version changes only when the canonical repository changes.

---

# Specification Version

Every specification document owns an independent version.

Example

```text
HYP-SPEC-010500

Version

1.0
```

Future

```text
1.1

1.2

2.0
```

---

# Object Version

Every Knowledge Object contains

```json
{
"version":"1.0"
}
```

The version belongs to the object.

Not to the file.

---

# Revision

Every accepted modification creates a new Revision.

Example

```text
HYP-LEM-000145

Version

1.0

↓

Revision

1

↓

Revision

2

↓

Revision

3
```

Identity never changes.

Revision always increases.

---

# Card Version

Cards may evolve.

Example

```text
Lemma Card

v1.0

↓

v1.1

↓

v1.2
```

The Card ID remains identical.

---

# Dataset Version

Training datasets are immutable snapshots.

Example

```text
Training Dataset

2027.01

↓

2027.02

↓

2027.03
```

Datasets are never edited.

New datasets are generated.

---

# Builder Version

Builders evolve independently.

Example

```text
Hypatia Builder

v1.0

↓

v1.1

↓

v2.0
```

Builder updates never change Repository history.

---

# Resolver Version

Resolvers own their own lifecycle.

Example

```text
Lemma Resolver

v1.3
```

A resolver update may improve linking without changing repository data.

---

# Knowledge Graph Version

Every graph generation produces a new graph version.

The graph is reproducible.

The repository remains unchanged.

---

# Website Version

Presentation evolves independently.

Example

```text
Website

v2.4
```

Website changes never affect scientific knowledge.

---

# AI Model Version

Models are versioned independently.

Example

```text
Hypatia Language Model

v0.9

↓

v1.0

↓

v2.0
```

Model updates never alter repository knowledge.

---

# Semantic Versioning

Hypatia follows Semantic Versioning.

```text
Major.Minor.Patch
```

---

## Major

Scientific incompatibility.

Repository redesign.

Schema redesign.

---

## Minor

New capabilities.

Additional fields.

New Resolver.

New Builder.

---

## Patch

Corrections.

Bug fixes.

Documentation improvements.

Performance improvements.

---

# Scientific Rule

Corrections never rewrite history.

They create a newer version.

---

# Version History

Every object stores

```json
{
"version":"1.3",
"revision":5,
"created_at":"",
"updated_at":""
}
```

The complete revision history remains accessible.

---

# Deprecation

Objects are never deleted.

Deprecated objects receive

```json
{
"status":"deprecated"
}
```

History remains intact.

---

# Archive

Archived objects retain

ID

Version

Revision

History

Relationships

Sources

Only their active status changes.

---

# Release

A Platform Release represents a stable scientific milestone.

Example

```text
Hypatia Founder Release

1.0
```

---

# Version Independence

Changing

Website

does not change

Repository Version.

Changing

AI Model

does not change

Knowledge Version.

Changing

Documentation

does not change

Object Version.

Every subsystem owns its own evolution.

---

# Time Travel

Hypatia must always allow reconstruction of any previous scientific state.

Every published version must remain reproducible.

---

# Version Motto

> **Nothing Is Lost.**

> **Everything Evolves.**

> **History Is Permanent.**

---

# Founder Statement

Knowledge is never finished.

It continuously approaches better understanding.

Versioning is therefore not merely a software mechanism.

It is the historical memory of scientific progress.

---

# Architectural Law

Every accepted scientific change produces a newer version.

No accepted scientific state may ever disappear.

History is considered part of knowledge.

---

# Academic Notes

Hypatia distinguishes clearly between:

* **Identity** (what the object is)
* **Revision** (how many times it has changed)
* **Version** (the scientific state of the object)
* **Release** (the public state of the platform)

This separation ensures that scholarly citations remain stable, repository snapshots remain reproducible, AI training datasets remain traceable, and every published conclusion can always be traced back to the exact state of knowledge from which it was derived.
