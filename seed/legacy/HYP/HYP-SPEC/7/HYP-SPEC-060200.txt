# Developer Guide

**Project:** Hypatia Knowledge Platform

**Document ID:** HYP-SPEC-060200

**Version:** 1.0 Founder Edition

**Status:** Official

**Classification:** Engineering Manual

**Parent Specification:** HYP-SPEC-060000 — Engineering Overview

---

# Purpose

This guide defines the engineering standards, coding philosophy, development workflow, and implementation practices required for software developers contributing to the Hypatia Platform.

It ensures that all software development remains aligned with the architectural, scientific, and engineering principles established by the official specifications.

---

# Philosophy

Developers implement features.

Builders construct systems.

Specifications govern both.

---

# Fundamental Principle

Every line of code must have a corresponding architectural purpose.

Code exists to realize specifications—not to redefine them.

---

# Developer Role

A Developer is responsible for

* implementing application logic
* developing user interfaces
* integrating platform services
* consuming APIs
* fixing defects
* improving performance
* maintaining code quality

Developers do **not** define scientific knowledge or architectural policy.

---

# Development Workflow

```text id="developer_workflow"
Read Specification

↓

Understand Existing Architecture

↓

Design Implementation

↓

Write Code

↓

Run Tests

↓

Review

↓

Commit

↓

Deploy
```

Every feature begins with the relevant specification.

---

# Development Principles

---

## Specification First

Implementation begins only after identifying the governing specification.

---

## Architecture First

Developers must understand the surrounding architecture before modifying any component.

---

## Reuse Before Creation

Existing services, utilities, Builders, and APIs should be reused whenever possible.

Duplicate functionality is discouraged.

---

## Small Features

Each implementation should solve one engineering objective.

Large unreviewable changes are prohibited.

---

## Documentation Synchronization

Whenever implementation changes,

documentation must be updated.

---

# Repository Awareness

Developers must understand the Repository structure but should interact with it only through approved services.

Direct manipulation of Repository files is prohibited unless explicitly authorized.

---

# Folder Discipline

Developers must follow

* Repository Structure
* Folder Standards
* File Naming Convention
* ID Convention

No arbitrary folder creation is permitted.

---

# Coding Standards

All code should be

* readable
* modular
* deterministic
* testable
* documented

Complexity should always be minimized.

---

# Dependency Policy

Developers should

* minimize external dependencies
* avoid unnecessary frameworks
* document every dependency
* preserve portability

Every dependency must have a documented justification.

---

# Error Handling

Applications must

* detect failures
* return meaningful errors
* avoid silent exceptions
* preserve user data
* generate diagnostic logs

---

# Logging

Application logs should record

```text id="developer_logging"
module

operation

duration

status

user

error_code

timestamp
```

Sensitive information must never be logged.

---

# API Usage

Developers interact with platform functionality through official APIs.

Business logic must not be duplicated inside applications.

---

# Configuration

Configuration values belong in configuration files or environment variables.

Hardcoded configuration is prohibited.

---

# Testing Requirements

Every feature must include

* Unit Tests
* Integration Tests
* Regression Tests

Bug fixes require regression tests whenever applicable.

---

# Code Review

Every change must undergo review before integration.

Review criteria include

* specification compliance
* architecture consistency
* readability
* performance
* security
* documentation

---

# Security

Developers must

* validate all inputs
* sanitize outputs
* avoid exposing secrets
* use least-privilege access
* follow Security Specification

Security is considered a development responsibility.

---

# Performance

Optimization should never compromise

* correctness
* maintainability
* explainability

Readable code has priority unless measurable performance requires otherwise.

---

# Version Control

Every commit should

* address one objective
* reference specifications
* include descriptive messages

History must remain understandable.

---

# Developer Success Criteria

A successful Developer

* follows specifications
* respects architecture
* writes maintainable code
* tests thoroughly
* documents changes
* collaborates effectively

---

# Long-Term Objective

To cultivate a disciplined development culture in which software remains maintainable, modular, secure, and permanently aligned with the scientific mission of the Hypatia Platform.

---

# Scientific Law

Specifications guide Architecture.

Architecture guides Development.

Development serves Knowledge.

---

# Developer Motto

> **Write Less. Build Better. Respect the Architecture.**

---

# Founder Statement

Within Hypatia, software development is not measured by the amount of code produced.

It is measured by how faithfully that code embodies the architectural vision and scientific objectives of the platform.

Developers are therefore encouraged to think first as engineers, second as programmers, and always as custodians of a long-lived institutional knowledge system.

---

# Academic Notes

The Developer Guide adopts a **Specification-Governed Development (SGD)** methodology.

Under SGD, software engineering is constrained by formal architectural specifications, ensuring that implementation remains reproducible, reviewable, and scientifically aligned.

This methodology reduces architectural drift, improves maintainability, and enables large teams to collaborate without compromising the integrity of the underlying knowledge platform.
