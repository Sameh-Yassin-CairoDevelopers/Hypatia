# Purpose

This guide defines the engineering standards, workflows, validation rules, and operational procedures governing the import of external knowledge into the Hypatia Platform.

Importing does not insert data directly into the Repository.

It transforms external information into structured candidates that must pass the complete scientific governance process.

---

# Philosophy

Data is imported.

Knowledge is constructed.

Nothing enters the Repository without scientific processing.

---

# Fundamental Principle

Every imported item is considered **unverified** until it successfully completes the full validation lifecycle.

Import never bypasses governance.

---

# Import Architecture

```text id="import_architecture"
External Source

↓

Import Builder

↓

Normalization

↓

Validation

↓

Proposal Objects

↓

Review

↓

Repository
```

No external dataset reaches the Repository directly.

# Import Stages

Every import follows the same lifecycle.

```text id="import_flow"
Read

↓

Parse

↓

Normalize

↓

Validate

↓

Resolve

↓

Generate Proposal Objects

↓

```

No stage may be skipped.

# Normalization

Normalization includes

* Unicode normalization
* encoding verification
* whitespace normalization
* punctuation cleanup
* language normalization
* identifier normalization

Normalization must be deterministic.

---

# Validation

Validation checks

* schema compliance
* required fields
* identifier uniqueness
* language consistency
* reference integrity
* source completeness

Invalid records never enter the Proposal Queue.

---

# Resolver Integration

After validation,

every imported object passes through the Resolver.

Resolver responsibilities include

* duplicate detection
* canonical matching
* entity matching
* lemma matching
* relation discovery

Resolver decides whether

* create new object
* update existing object
* merge
* reject

---

# Proposal Generation

Every imported object becomes

* Dictionary Proposal
* Entity Proposal
* Story Proposal
* Grammar Proposal
* Relation Proposal
* Source Proposal

The Repository never receives raw imported data.

# Founder Statement

Within Hypatia, importing is not regarded as a technical operation.

It is the beginning of a scientific journey.

Every imported inscription, lexical entry, grammatical observation, or historical reference must travel through the same institutional pathway before becoming part of the permanent Repository.

This discipline ensures that the Repository remains an authoritative scholarly foundation rather than a passive collection of imported data.

---

# Academic Notes

The Hypatia Import Framework implements a **Knowledge Ingestion Pipeline (KIP)**.

Unlike conventional ETL systems that transfer records directly into operational databases, KIP transforms external information into governed Proposal Objects that undergo normalization, semantic resolution, academic review, and institutional validation before Repository integration.

This architecture guarantees provenance preservation, deterministic processing, scientific accountability, and long-term knowledge quality.

## ✦ Founder Insight

أقترح ألا يكون لدينا مستقبلًا **Import Builder** واحد فقط، بل **Import Adapter Framework**.

أى أن كل مصدر علمى كبير (مثل TLA، PharaLex، Bibliotheca Alexandrina، وغيرها) يمتلك **Adapter** خاصًا به يترجم بياناته إلى النموذج الداخلى الموحد لهيباتيا.

وبذلك يصبح المسار كالتالى:

```text
External Source
        ↓
Source Adapter
        ↓
Import Builder
        ↓
Normalization
        ↓
Resolver
        ↓
Proposal Objects
```

وبهذا يمكن إضافة مصدر جديد بالكامل دون تعديل أى جزء من النظام، بل فقط بإضافة **Adapter** جديد، وهو امتداد مباشر لفلسفة **Hypatia Builder Framework** و **Auto Build لكل شيء**.
